# 头脑风暴：从想法到设计

将想法通过自然的协作对话转化为完整的设计方案。

**核心原则：在任何实现代码之前，必须先完成设计并获得用户批准。** 无论项目多简单，都要走这个流程。

## ⚠️ 反模式："这个太简单不需要设计"

每个项目都必须经过此流程。一个 todo list、一个单函数工具、一个配置变更——都要。简单项目恰恰是最容易因未检验的假设导致浪费工作的地方。

## 检查清单

你必须为每个条目创建 TodoWrite 任务，并按顺序完成：

1. **探索项目上下文** — 检查文件、文档、最近提交、worklog.md
2. **逐个提问澄清** — 一次一个问题，理解目的/约束/成功标准
3. **提出 2-3 个方案** — 附带权衡分析和你的推荐
4. **分段呈现设计** — 按复杂度缩放，每段获得用户确认
5. **编写设计文档** — 保存到 `docs/plans/YYYY-MM-DD-<name>-design.md`
6. **设计自审** — 检查占位符、矛盾、歧义、范围蔓延
7. **用户审查设计文档** — 让用户确认后再进入实施
8. **过渡到实施** — 调用 sp-writing-plans 技能创建实施计划

## 流程

```
探索项目上下文 → 逐个提问 → 提出2-3方案 → 分段呈现设计
                                                  ↓
                                            用户批准？
                                           ↙        ↘
                                         是          否 → 回到提问
                                          ↓
                                    编写设计文档
                                          ↓
                                      设计自审
                                          ↓
                                    用户审查确认
                                          ↓
                              → sp-writing-plans → sp-executing-plans
```

## 澄清问题模板

逐个提出以下类型的问题（不要一次问多个）：

- **目的**："这个功能要解决什么问题？"
- **用户**："谁会使用这个功能？"
- **输入/输出**："期望的输入和输出是什么？"
- **约束**："有什么技术或业务限制？"
- **成功标准**："怎么判断做成了？"
- **优先级**："哪些功能是必须有，哪些可以后续加？"

## 设计文档模板

```markdown
# [功能名称] 设计文档

**日期：** YYYY-MM-DD
**状态：** 草稿 / 待审查 / 已批准

## 目标
一句话描述要构建什么。

## 方案选择
### 方案 A: [名称]
- 描述：...
- 优势：...
- 劣势：...

### 方案 B: [名称]（推荐）
- 描述：...
- 优势：...
- 劣势：...

### 方案 C: [名称]
- 描述：...
- 优势：...
- 劣势：...

## 最终设计
### 架构
[2-3 句话描述技术方案]

### 数据模型
[关键数据结构]

### API / 接口
[关键接口定义]

### 文件结构
[需要创建/修改的文件]

## 非目标
明确列出不做的事情，防止范围蔓延。

## 风险与缓解
| 风险 | 缓解措施 |
|------|---------|
```

## 铁律

1. **不经过设计确认，不写任何代码**
2. **一个问题一个问题地问**，不要一次抛出多个
3. **设计文档必须包含"非目标"** 防止范围蔓延
4. **用户有最终决定权**，推荐方案但不替用户决定
