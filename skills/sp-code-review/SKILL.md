# 代码审查

## 概述

派遣独立的代码审查子代理，在问题扩散之前发现它们。审查者获得精确的上下文进行评估——而不是你的会话历史。这保持审查者专注于工作产品，并保留你自己的上下文继续工作。

**核心原则：早审查，常审查。**

## 何时请求审查

**必须审查：**
- 完成每个主要任务后
- 实现重大功能后
- 合并到主分支前

**建议审查：**
- 卡住时（获得新视角）
- 重构前（基线检查）
- 修复复杂 bug 后

## 如何请求审查

### 1. 获取变更范围

```bash
# 获取变更的文件列表
git diff --name-only HEAD~1
git diff --stat HEAD~1
```

### 2. 派遣审查子代理

使用 Task 工具，提供以下信息：

```
请对以下代码进行审查：

## 实现内容
[描述你刚刚构建了什么]

## 需求/计划
[它应该做什么，引用设计文档或计划]

## 变更范围
[变更的文件列表和关键代码片段]

## 重点检查
- 逻辑正确性
- 边界条件处理
- 错误处理
- 代码风格一致性
- 测试覆盖
- 安全问题
```

### 3. 根据反馈行动

| 严重度 | 行动 |
|--------|------|
| 🔴 Critical | 立即修复 |
| 🟡 Important | 继续前修复 |
| 🔵 Minor | 记录稍后处理 |
| 💡 Suggestion | 考虑采纳 |

## 审查检查清单

审查子代理应检查：

**正确性：**
- [ ] 逻辑是否正确
- [ ] 边界条件是否处理
- [ ] 错误/异常是否处理
- [ ] 数据类型是否匹配

**代码质量：**
- [ ] 命名是否清晰
- [ ] 函数是否简洁（单一职责）
- [ ] 是否有重复代码
- [ ] 注释是否必要且准确

**安全性：**
- [ ] 是否有注入风险
- [ ] 敏感数据是否保护
- [ ] 输入是否验证

**测试：**
- [ ] 测试是否覆盖关键路径
- [ ] 测试是否可读
- [ ] 边界情况是否测试

## 审查结果格式

```markdown
## 审查结果

### 优点
- [列出做得好的地方]

### 问题

#### 🔴 Critical
- [必须立即修复的问题]

#### 🟡 Important
- [应该修复的问题]

#### 🔵 Minor
- [可选改进]

### 评估
[整体判断：可以继续 / 需要修改后重新审查]
```

## 铁律

1. **不要因为"太简单"跳过审查**
2. **不要忽略 Critical 问题继续推进**
3. **如果有技术理由，可以驳回审查意见**（附证据）
