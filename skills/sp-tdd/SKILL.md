# 测试驱动开发 (TDD)

## 概述

先写测试，看着它失败，再写最小代码让它通过。

**核心原则：如果你没看到测试失败，你就不知道它测试了什么。**

## 铁律

```
没有失败的测试，就不写任何生产代码。
```

先写了代码再补测试？删掉它，重来。

- 不要保留作为"参考"
- 不要在写测试时"借用"它
- 不要看它
- 删除就是删除

从测试重新实现。没有例外。

## 何时使用

**必须使用：**
- 新功能
- Bug 修复
- 重构
- 行为变更

**可跳过（需征得用户同意）：**
- 一次性原型
- 配置文件
- 纯展示型代码

## 红-绿-重构循环

```
🔴 RED（红）→ 🟢 GREEN（绿）→ 🔵 REFACTOR（重构）
     ↑                                  |
     └────────── 下一个测试 ←───────────┘
```

### 🔴 RED — 写失败测试

写一个最小测试来展示期望行为。

**要求：**
- 测试一个行为
- 名称清晰（描述期望，不是描述实现）
- 用真实代码（不 mock，除非不可避免）

示例：
```python
# ✅ 好的测试名
def test_retry_returns_success_after_3_attempts():
    ...

# ❌ 糟糕的测试名
def test_retry():
    ...
```

### ✅ 验证 RED — 看它失败

**必须执行。永远不要跳过。**

确认：
- 测试失败（不是报错）
- 失败信息符合预期
- 失败是因为功能缺失（不是拼写错误）

**测试通过了？** 你在测试已有行为，修改测试。
**测试报错了？** 修复错误，重新运行直到正确失败。

### 🟢 GREEN — 最小代码

写最简单的代码让测试通过。

```python
# 不求优雅，只求通过
def retry_operation(fn, max_retries=3):
    for _ in range(max_retries):
        try:
            return fn()
        except Exception:
            continue
    raise Exception(f"Failed after {max_retries} attempts")
```

**规则：**
- 不要过度设计
- 不要添加"以后会用"的功能
- 可以硬编码（重构阶段再泛化）
- 消除重复即可，不求完美

### 🔵 REFACTOR — 清理

测试通过后，安全地改进代码。

**可以做的：**
- 提取函数/变量
- 改善命名
- 简化逻辑
- 消除重复
- 优化性能

**必须确保：**
- 每次小改动后重新运行测试
- 所有测试持续通过
- 行为不变

## TDD 步骤检查清单

```
□ 写一个失败的测试
□ 运行测试，确认失败原因正确
□ 写最小代码让测试通过
□ 运行测试，确认通过
□ 如果需要，重构
□ 运行所有测试，确认全部通过
□ 提交
```

## 反模式

- ❌ "先写个框架代码，稍后补测试" — 稍后永远不会来
- ❌ "这个太简单不需要测试" — 简单代码也有 bug
- ❌ "测试覆盖率够了就行" — 覆盖率 ≠ 质量
- ❌ 写完一个功能再补所有测试 — 这不是 TDD
