# 系统化调试

## 概述

随机修复浪费时间并制造新 bug。快速补丁掩盖底层问题。

**核心原则：在尝试修复之前，必须找到根因。症状修复就是失败。**

## 铁律

```
没有根因调查，不提出任何修复方案。
```

如果你还没完成第一阶段，就不能提出修复。

## 何时使用

任何技术问题：
- 测试失败
- 生产环境 Bug
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**以下情况更要使用：**
- 时间紧迫时（紧急情况让人想猜测）
- "就一个快速修复"看起来很明显时
- 你已经尝试了多个修复方案
- 上一个修复没起作用
- 你没完全理解问题

## 四个阶段

你必须按顺序完成每个阶段。

### 第一阶段：根因调查

**在尝试任何修复之前：**

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 它们通常包含确切的解决方案
   - 完整阅读堆栈追踪
   - 记录行号、文件路径、错误代码

2. **稳定复现**
   - 你能可靠地触发它吗？
   - 确切的步骤是什么？
   - 每次都发生吗？
   - 如果不稳定 → 收集更多数据，不猜测

3. **检查最近的变更**
   - 什么变更可能导致这个问题？
   - Git diff、最近提交
   - 新依赖、配置变更
   - 环境差异

4. **多组件系统中收集证据**
   在每个组件边界添加诊断日志：
   - 记录进入组件的数据
   - 记录退出组件的数据
   - 验证环境/配置传播
   - 检查每层的状态

5. **缩小范围**
   - 二分法定位问题区域
   - 注释代码排除法
   - 添加断言验证假设

### 第二阶段：确认根因

在提出修复之前，明确表述：

```
根因：[具体描述]
证据：[支持这一结论的数据]
影响范围：[哪些地方受影响]
```

### 第三阶段：制定修复方案

1. 设计修复（不只是补丁）
2. 考虑副作用
3. 考虑是否需要回归测试
4. 如果修复范围大 → 先创建计划

### 第四阶段：修复并验证

1. 实施修复
2. 写回归测试（确保同类问题不再发生）
3. 运行完整测试套件
4. 验证原始问题已解决
5. 检查是否引入新问题

## 调试日志模板

```python
# 在关键点添加诊断
logger.info(f"[DEBUG] Step 1: input={input_data}, type={type(input_data)}")
# ... 处理 ...
logger.info(f"[DEBUG] Step 2: after_transform={result}, errors={errors}")
```

## 何时求助

**如果超过 15 分钟没找到线索：**
- 停下来，重新审视错误信息
- 尝试最小可复现示例
- 向用户描述你做了什么、排除了什么

## 反模式

- ❌ "这个修改应该能修复" — 没有根因就改代码
- ❌ 改了一个地方不行，再改另一个 — 盲目试错
- ❌ "先注释掉这个试试" — 治标不治本
- ❌ 修复一个问题引入两个新问题 — 没考虑副作用
