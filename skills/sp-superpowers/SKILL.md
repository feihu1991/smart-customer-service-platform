# Superpowers — 结构化开发工作流

## 概述

Superpowers 是一套为 AI 编程代理设计的结构化开发流程框架。它强制走"构思→计划→执行→验证"的完整流程，而不是让 AI 直接开写。

**核心理念：约束产生质量。**

## 技能体系

```
sp-superpowers (调度入口)
    ├── sp-brainstorming    → 头脑风暴，构思设计
    ├── writing-plans       → 编写实施计划
    ├── sp-executing-plans  → 按计划执行
    ├── sp-tdd              → 测试驱动开发
    ├── sp-debugging        → 系统化调试
    └── sp-code-review      → 代码审查
```

## 何时使用

- 任何需要写代码的任务
- 任何需要多步骤实现的功能
- 任何需要系统化处理的 bug

## 完整工作流

```
1. sp-brainstorming（构思）
   ├── 探索项目上下文
   ├── 逐个提问澄清需求
   ├── 提出 2-3 个方案
   ├── 呈现设计并获得批准
   └── 编写设计文档
         ↓
2. writing-plans（计划）
   ├── 将设计分解为细粒度任务
   ├── 每个任务 2-5 分钟可完成
   ├── 包含完整代码和验证步骤
   └── 保存到 docs/plans/
         ↓
3. sp-executing-plans（执行）
   ├── 加载并审查计划
   ├── 逐步执行每个任务
   ├── 每步都运行验证
   └── 更新 worklog.md
         ↓
4. sp-tdd（贯穿全程）
   ├── 先写失败测试
   ├── 看它失败
   ├── 写最小代码通过
   └── 重构
         ↓
5. sp-debugging（遇到问题时）
   ├── 四阶段根因分析
   ├── 确认根因
   ├── 制定修复方案
   └── 修复并验证
         ↓
6. sp-code-review（关键节点）
   ├── 完成主要任务后
   ├── 派遣子代理审查
   ├── 根据反馈修改
   └── 最终确认
```

## 快速启动

根据任务复杂度选择流程：

### 🟢 简单任务（< 30 分钟）
```
直接实现 → 跑测试 → 完成
```
适合：修个 typo、改个配置、加个简单字段

### 🟡 中等任务（30 分钟 - 2 小时）
```
sp-brainstorming(简化版) → writing-plans → 执行 → sp-code-review
```
适合：新增功能、修改业务逻辑、重构

### 🔴 复杂任务（> 2 小时）
```
完整流程：sp-brainstorming → writing-plans → sp-executing-plans → sp-tdd → sp-code-review
```
适合：新系统、架构变更、多组件协作

## 使用方式

### 方式一：从 sp-superpowers 入口
调用 sp-superpowers，它会根据任务复杂度自动选择合适的流程和子技能。

### 方式二：直接调用子技能
如果你已经知道需要哪个具体技能，可以直接调用：
- `sp-brainstorming` — 只需要头脑风暴
- `writing-plans` — 已有设计，需要写计划
- `sp-executing-plans` — 已有计划，需要执行
- `sp-tdd` — 只想用 TDD 方式开发
- `sp-debugging` — 遇到 bug 需要调试
- `sp-code-review` — 需要代码审查

## 指令优先级

1. **用户直接指令** — 最高优先级
2. **Superpowers 技能** — 覆盖默认行为
3. **默认系统行为** — 最低优先级

用户说"不用 TDD"，就听用户的。用户是老大。

## 铁律

1. **设计先于代码** — 不经过思考直接写代码是最大的浪费
2. **计划先于执行** — 没有计划的执行是盲目的
3. **测试先于实现** — TDD 不是选项，是纪律
4. **根因先于修复** — 猜测式调试只会制造更多问题
5. **审查先于合并** — 第二双眼睛能发现你忽略的
6. **用户是最终决策者** — 所有推荐都要用户确认
